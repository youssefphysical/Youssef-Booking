1. SESSION LOCATION MODEL

---

Current architecture treats gym/area/location as optional.
That is NOT correct for session-related emails.

The app is currently a solo-trainer model, not a multi-gym platform.
However:
session-related bookings WILL contain a selectable training location.

Create a lightweight “Booking Location Model” for session-related events only.

Session-related emails MUST support:

- city
- area
- gym name
- full location label
- optional map URL

But ONLY for:

- session confirmed
- session reminder
- session rescheduled
- session cancelled
- no-show
- admin session booking alerts

Do NOT apply location logic globally to all emails.

Document:

- which future fields are needed
- where they should live logically
- how the system should behave if a location is missing

Do NOT implement schema changes yet.
Only document the architecture correctly.
