ADMIN CONTROL PANEL FINAL REORGANIZATION + DESIGN SYSTEM + MOBILE UX + PERFORMANCE HARDENING

IMPORTANT EXECUTION RULES:

This is NOT a full redesign.
This is NOT a new feature task.
This is NOT a business-logic change.

Implement this plan incrementally in small phases.
After each phase, verify that the app still runs and report a short checkpoint.
Do not continue blindly if a risky issue appears.

Do NOT:
- change booking rules
- change package/session logic
- change payment logic
- change authentication rules
- delete user/client data
- reduce image quality
- compress public images in a way that lowers quality
- remove existing functionality
- add new features beyond this plan
- add unnecessary animations or visual clutter
- increase Neon/database transfer
- reintroduce base64 image payloads

Goal:
Make the Admin Control Panel cleaner, easier to manage on mobile, safer, more consistent, lighter on hosting/Neon, and more professional while preserving all existing functionality.

====================================================
PHASE 1 — ADMIN NAVIGATION RESTRUCTURE
====================================================

Simplify the primary admin navigation to only daily-use tools.

Primary navigation should contain:

1. Dashboard
2. Clients
3. Bookings
4. Media
5. More

Move non-daily tools into More.

The More page should be grouped clearly:

Business:
- Analytics
- Leads
- Sessions & Packages

Operations:
- Command Center
- Recovery Center

System:
- Settings
- Audit Log
- Integrity Checker
- System Health

Advanced Tools:
- Merge Duplicate Clients
- Maintenance Tools
- Repair Expired Sessions
- Developer/System Utilities

Rules:
- Do not delete any tool.
- Only move and organize.
- Advanced/dangerous tools must not appear in the primary daily navigation.
- The More page must not become a messy long list.
- Each More item should be a compact row/card with:
  - icon
  - title
  - short description
  - optional badge if attention is needed

====================================================
PHASE 2 — ADMIN DESIGN SYSTEM STANDARDIZATION
====================================================

Apply one consistent visual system across the entire Admin Control Panel.

Keep the existing Tron Legacy / premium dark theme.
Do not redesign the visual identity.
Just standardize the layout and UI behavior.

PAGE STRUCTURE STANDARD:

Every admin page should follow this structure:

- Small uppercase section label
- Clear page title
- Short description, maximum 1–2 lines
- Main content cards

Long explanations must be collapsed behind:
- Info button
- Details toggle
- Help tooltip

Do not allow long descriptions to consume large vertical space on mobile.

GLOBAL SPACING SYSTEM:

Mobile:
- Page horizontal padding: 20px
- Card padding: 20px
- Section gap: 20px
- Card gap: 16px
- Bottom spacer: at least 120px

Tablet:
- Page horizontal padding: 24px
- Card padding: 22px

Desktop:
- Page horizontal padding: 32px
- Card padding: 24px
- Max content width: 1200px

Rules:
- No page title should be clipped from the left or right.
- No content should touch screen edges.
- Spacing should feel like a native mobile app.

CARD SYSTEM:

Standardize all admin cards:

- Border radius: 24px
- Consistent dark background
- Subtle border
- Subtle Tron cyan glow only for active/selected states
- No heavy glow on normal cards
- Consistent internal spacing
- Consistent header/title/body/action layout

Card structure:
- Header
- Optional description
- Content
- Actions

Do not mix unrelated card styles.

BUTTON SYSTEM:

Use only these button types:

Primary:
- Save
- Add
- Confirm main action

Secondary:
- Edit
- Open profile
- Reschedule
- View details

Ghost:
- Back
- Cancel
- More
- Expand

Danger:
- Delete
- Merge final
- Reset all
- Repair system actions

Rules:
- Dangerous actions must not use the same cyan style as safe actions.
- Dangerous actions must require confirmation.
- On mobile, crowded action rows should collapse into an Actions menu.

BADGE / STATUS SYSTEM:

Standardize all status badges:

Green:
- Active
- Paid
- Completed

Yellow:
- Pending
- Needs review
- Awaiting action

Red:
- Failed
- Expired
- Dangerous
- Delete

Cyan:
- New
- Live
- Selected

Gray:
- Draft
- Inactive
- Archived

Use the same status colors everywhere.

FORM FIELD SYSTEM:

Standardize inputs, selects, textareas, and filters:

- Height: 52px on mobile
- Border radius: 16px
- Label above the field
- Helper text below when needed
- Clear error state
- Consistent disabled state
- Consistent focus glow

Apply to:
- Leads filters
- Booking filters
- Package filters
- Settings fields
- Client search
- Merge client selector

MOBILE DROPDOWN RULE:

On mobile, any dropdown with more than 6 options should open as a bottom sheet.

Bottom sheet requirements:
- Search if options are many
- Clear selected option
- Cancel button
- Large touch targets
- Smooth close
- Does not overflow screen
- Does not hide behind browser UI or bottom nav

EMPTY STATE SYSTEM:

Standardize empty states.

Compact empty states for:
- No recovery requests
- No leads match
- No upcoming sessions
- No alerts
- No package verification requests
- No audit results

Empty states should include:
- Small icon
- Short title
- One short helper sentence
- Optional action button

Do not use oversized empty cards unless it is the main page content.

TABLE / LIST RESPONSIVENESS:

Tables should become cards on mobile.

Apply to:
- Audit Log
- Integrity results
- Analytics details
- Package lists
- Client lists if needed

Mobile cards should show:
- Main label
- Key metadata
- Status badge
- View details action

====================================================
PHASE 3 — GLOBAL MOBILE LAYOUT FIX
====================================================

Create or repair a consistent admin layout container.

Requirements:
- No page title should be cut from the left.
- No content should touch screen edges.
- Consistent side padding.
- Consistent top safe area.
- Consistent bottom safe area.
- Bottom spacer must protect against FAB + bottom nav overlap.
- Horizontal tab bars must scroll smoothly without clipping.
- Long descriptions collapse on mobile.
- Empty states should be compact.
- Dropdowns with many options must become bottom sheets on mobile.
- FAB must never cover important controls.
- Icon-only buttons must have accessible labels or tooltips.
- Touch targets must be at least 44px high.

Apply this globally to all admin pages.

PHASE 4 — TOP TAB BAR STANDARD
====================================================

Standardize all horizontal tab bars.

Requirements:
- Smooth horizontal scroll.
- No clipped tabs.
- Equal height.
- Equal padding.
- Clear active state.
- Subtle cyan glow for active tab only.
- Proper left/right safe padding.
- No tab should appear half-cut without scroll affordance.

Apply to:
- Media tabs
- Settings sub-tabs
- Bookings tabs
- Client-related tabs
- Any admin top tab system

PHASE 18 — SUPER ADMIN PROTECTION
====================================================

Dangerous tools must be Super Admin only:

- Merge Clients
- Maintenance Tools
- Repair Expired Sessions
- Integrity fixes
- Delete/destructive admin operations
- Developer/System Utilities

If the current user is not Super Admin:
- hide these tools from navigation
- block direct route access
- return unauthorized response from API

Keep the existing super admin identity/role system.

Do not weaken security.

====================================================
PHASE 19 — UNIFIED DANGER CONFIRMATION
====================================================

Create a consistent confirmation pattern for dangerous actions.

Danger confirmation should include:
- clear warning title
- short summary of what will happen
- affected records count if available
- typed confirmation keyword when needed:
  - MERGE
  - DELETE
  - REPAIR
  - RESET
- audit log entry

Apply this to:
- Merge Clients
- Delete Booking
- Reset Settings
- Repair Expired Sessions
- Delete Media
- Any destructive system action

Dangerous actions must not be easy to tap accidentally on mobile.

PHASE 22 — STANDARD PAGE STATES
====================================================

Every admin page must support exactly these states:

- Loading
- Empty
- Error
- Success

Never show contradictory states together.

Examples:
- Do not show “Failed to load integrity report” and “All clean” at the same time.
- Do not show a huge empty chart if there is no data.
- Do not show a large empty recovery page if there are zero requests.

====================================================
PHASE 23 — PERFORMANCE AND NEON PROTECTION
====================================================

Do not increase Neon transfer.

Rules:
- No image blobs in API responses.
- No base64 image payloads.
- No polling on hidden/inactive pages.
- No unnecessary refetchOnWindowFocus.
- Lazy load heavy admin sections.
- Paginate growing lists:
  - Clients
  - Leads
  - Bookings
  - Audit Log
  - Packages
  - Merge client search
- Keep response payloads small.
- Add development warning for API responses over 200 KB.
- Keep public image quality unchanged.
- Use thumbnails only inside admin previews/lists.
- Public website must always use original high-quality images.

Heavy sections should use:
- Last updated timestamp
- Manual refresh button
- No frequent auto-refresh
