CRITICAL TASK: Fix Neon network transfer overuse from the root cause.

Do NOT add new features.

Do NOT redesign the UI.

Do NOT modify unrelated business logic.

Goal:
Reduce Neon database network transfer dramatically by removing wasteful queries, removing image/blob transfer from PostgreSQL, reducing polling, and preventing duplicated API calls.

Current problem:
Neon Free monthly 5 GB transfer was exceeded. Audit shows the main causes are:

1. getAllClients() bulk query with profile pictures/blobs
2. /api/admin/analytics full scans
3. /api/admin/media sending hero images/settings blobs
4. /api/dashboard/stats duplicated scans
5. /api/settings sending logo/base64 blobs
6. excessive polling on bookings, feature flags, packages, alerts, command center, notifications

Implement the following fixes in this exact priority order.

==================================================
PRIORITY 1 — REMOVE IMAGES/BLOBS FROM DATABASE RESPONSES

PostgreSQL must NOT return heavy image blobs or base64 data in normal API responses.

Fix all APIs that return:

- profilePictureUrl blobs
- logo base64
- hero image base64
- splash image base64
- branding image blobs
- media blobs
- large JSON media payloads

Rules:

- Store only lightweight URLs, keys, IDs, metadata, dimensions, and timestamps in PostgreSQL.
- Do not include base64 image data in list endpoints.
- Do not send image blobs inside /api/settings, /api/admin/media, /api/admin/clients, /api/admin/analytics, or dashboard endpoints.
- Image files should be loaded by the browser from object storage/CDN/static file URL, not through Neon query payloads.

If object storage integration already exists, use it.
If not, create a clean abstraction layer:
mediaStorage.getUrl(key)
mediaStorage.save(file)
mediaStorage.delete(key)

Do not hardcode provider-specific logic inside UI components.

==================================================
PRIORITY 2 — FIX getAllClients()

getAllClients() is the biggest transfer consumer.

Change it immediately:

- Bulk client list must return summary fields only.
- Do NOT return profile pictures or heavy nested blobs.
- Do NOT return large notes/history/package/session data unless needed.
- Add pagination.
- Add search/filter parameters.
- Add sorting.
- Add a separate endpoint for single-client detail.

Example:

GET /api/admin/clients?page=1&pageSize=25&search=&sort=

Returns:

- id
- fullName
- phone
- email
- status
- activePackageSummary
- nextBookingAt
- lastActivityAt
- profilePictureThumbUrl only if it is a lightweight URL

Do NOT return full profile image blobs.

Create:

GET /api/admin/clients/:id

for full client detail only when opening a client profile.
